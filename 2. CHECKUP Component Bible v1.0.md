# CHECKUP Component Bible v1.0

## PART 01 — Component Architecture & Global Rules

# CB-001

Purpose

This document defines every reusable UI component used across the
CHECKUP website.

The purpose of the Component Bible is to guarantee:

- Consistency
- Reusability
- Maintainability
- Predictable UI behavior
- Scalable architecture

# CB-002

Component Philosophy

Every UI element must be treated as a reusable component.

No page may create its own version of an existing component.

# CB-003

Hierarchy

The component hierarchy is:

    Layout Components

    ↓

    Section Components

    ↓

    UI Components

    ↓

    Primitive Elements

# CB-004

Component Categories

The entire project consists of only four categories:

### Layout

Examples

- Header
- Footer
- Container
- Section
- Grid

### Sections

Examples

- Hero
- CTA Banner
- FAQ Section
- Timeline Section
- Framework Section

### Cards

Examples

- StatCard
- FeatureCard
- OpportunityCard
- FrameworkCard
- CaseStudyCard
- InsightCard

### UI

Examples

- Button
- Input
- Select
- Accordion
- Tabs
- Badge
- Modal
- Tooltip
- Spinner

# CB-005

Naming Convention

Every reusable component uses PascalCase.

Examples

    Hero

    HeroCompact

    SectionTitle

    FeatureCard

    FrameworkCard

    StatCard

    CTASection

# CB-006

Single Responsibility Rule

Every component should solve one problem only.

Example

Correct

    SectionTitle

Wrong

    SectionTitleWithButtonAndBackgroundAndStats

# CB-007

Content Rule

Components never contain business text.

Business content always comes from the content layer.

# CB-008

Responsive Rule

Every component must support:

- Desktop
- Tablet
- Mobile

without requiring a separate implementation.

# CB-009

Accessibility Rule

Every component must be fully keyboard accessible unless purely
decorative.

# CB-010

Animation Rule

Animations belong to the component itself.

Pages should not redefine component animations.

# CB-011

Color Rule

Components never introduce new colors.

Only colors defined by the global design tokens may be used.

# CB-012

Spacing Rule

Components never define arbitrary spacing.

They use the global spacing scale.

# CB-013

Component States

Every interactive component must explicitly support:

- Default
- Hover
- Focus
- Active
- Disabled

If applicable:

- Loading
- Success
- Error

# CB-014

Composition Rule

Complex components should be assembled from smaller reusable components.

Example

Hero

contains

- Section
- Container
- Heading
- Paragraph
- Button
- Illustration

rather than implementing everything internally.

# CB-015

Definition of a Shared Component

A component becomes “shared” if it appears on two or more pages.

It must then exist only once in the codebase.

# CB-016

Version 1 Component Inventory

The initial project consists of approximately:

- 8 Layout Components
- 12 Section Components
- 15 Card Components
- 20 UI Components

All remaining pages must reuse these components instead of introducing
new variants unless technically necessary.

## PART 11 — Reusable UX Patterns

# UX-001 — Purpose

This section defines reusable interaction patterns across the entire
website.

Users should never have to relearn how to interact with a new page.

Interaction consistency is a trust signal.

# UX-002 — Scanning Pattern

Every page should support the natural executive scanning behavior.

Expected reading sequence:

    Headline

    ↓

    Visual Cue

    ↓

    Key Insight

    ↓

    Supporting Information

    ↓

    Action

Never force users to read long paragraphs before understanding the page.

# UX-003 — Progressive Disclosure

Reveal information gradually.

Never expose all information simultaneously.

Pattern

    Overview

    ↓

    Details

    ↓

    Deep Information

    ↓

    Action

Examples

- FAQ
- Process
- Frameworks
- Opportunity Paths

# UX-004 — Information Density

Every viewport should contain one dominant message.

Avoid multiple competing focal points.

# UX-005 — Visual Hierarchy

Priority order:

1.  Headline
2.  Primary CTA
3.  Diagram / Illustration
4.  Supporting Text
5.  Secondary Actions

# UX-006 — Reading Length

Long paragraphs should rarely exceed

5 lines on desktop

6 lines on mobile.

Break long content into smaller logical blocks.

# UX-007 — CTA Hierarchy

Every screen should have only one Primary CTA.

Secondary actions must never visually compete with the primary action.

# UX-008 — Interaction Feedback

Every interaction provides immediate feedback.

Examples

Hover

↓

Visual response

Click

↓

Pressed state

Submit

↓

Loading

Success

↓

Confirmation

No interaction should appear “dead”.

# UX-009 — Hover Philosophy

Hover effects communicate interactivity.

They should never become decoration.

Allowed

- Shadow
- Border
- Slight color transition

Forbidden

- Large scaling
- Rotation
- Elastic movement
- Flashing

# UX-010 — Scroll Behaviour

Scrolling should feel continuous.

Avoid abrupt visual interruptions between sections.

# UX-011 — Trust Pattern

Every major page should naturally answer:

Who are you?

↓

What do you do?

↓

Why should I trust you?

↓

How do we work?

↓

How do I start?

This order should emerge through layout rather than explicit labeling.

# UX-012 — Empty States

If no content exists,

display:

Illustration

↓

Title

↓

Explanation

↓

Recommended Action

Never leave blank areas.

# UX-013 — Error States

Errors should explain:

- what happened
- why it happened
- what to do next

Never display technical error messages to users.

# UX-014 — Success States

After successful actions,

always explain the next expected step.

Never leave users uncertain about what happens next.

# UX-015 — Microcopy

Microcopy should be:

Short

Direct

Confident

Professional

Never humorous.

Never overly emotional.

# UX-016 — Decision Fatigue

Avoid presenting too many equally weighted choices.

Where possible,

guide users toward a single recommended action.

# UX-017 — Consistency

The same interaction must always produce the same result throughout the
website.

Buttons, cards, forms and navigation should never change behavior
between pages.

# UX-018 — Mobile Experience

Mobile is not a reduced desktop.

Layouts should be intentionally redesigned for vertical reading while
preserving the same information hierarchy.

# UX-019 — Accessibility

All UX patterns must remain usable with:

- keyboard navigation
- screen readers
- reduced motion preferences

Accessibility is part of the UX, not an optional enhancement.

# UX-020 — Acceptance Criteria

Reusable UX patterns are accepted only if:

- Interaction behavior is consistent across all pages.
- Information hierarchy remains predictable.
- Primary actions are always obvious.
- Progressive disclosure is used where appropriate.
- Empty, loading, success and error states are fully designed.
- Users never need to learn a new interaction pattern when navigating
  between pages.

## PART 12 — Component Quality Standards

# QS-001 — Purpose

This section defines the engineering quality requirements for every
reusable component.

These are mandatory implementation standards.

# QS-002

Every component must be:

- Reusable
- Stateless whenever possible
- Predictable
- Independent
- Easily testable

# QS-003

A component should solve exactly one UI responsibility.

If a component starts solving multiple unrelated problems, it must be
split.

# QS-004

Components should receive all configurable values through Props.

Never hardcode:

- titles
- descriptions
- labels
- URLs
- icons
- colors
- spacing
- page-specific values

# QS-005

Business logic does not belong inside UI components.

Components render.

Pages decide.

# QS-006

Every component must have sensible default props.

It should render safely even when optional props are omitted.

# QS-007

Components should avoid internal state whenever derived state can be
used.

Prefer controlled components over uncontrolled ones.

# QS-008

A component must never directly access routing, global data or APIs
unless it is explicitly designed for that purpose.

# QS-009

Every component should be composable.

Example

Correct

\`\`\`text id=“component_compose”

    <CardHeader />

    <CardBody />

    <CardFooter />


    Avoid monolithic implementations.

    ---

    # QS-010

    Avoid prop explosion.

    If a component requires an excessive number of props, reconsider its design.

    ---

    # QS-011

    Children should be used where flexibility is required.

    Do not duplicate components solely because of content differences.

    ---

    # QS-012

    Every component must gracefully handle missing optional content.

    Examples

    No illustration

    ↓

    Layout remains correct.

    No subtitle

    ↓

    Spacing remains balanced.

    ---

    # QS-013

    Every interactive component must expose a disabled state.

    No exceptions.

    ---

    # QS-014

    Component styling must remain independent of page context.

    A component should look identical regardless of where it is rendered.

    ---

    # QS-015

    Responsive behavior belongs to the component itself.

    Pages should not implement responsive fixes for individual components.

    ---

    # QS-016

    Components should avoid unnecessary re-rendering.

    Implementation should favor efficient rendering patterns.

    ---

    # QS-017

    Icons, buttons, typography and spacing should always come from shared primitives.

    Never recreate them locally.

    ---

    # QS-018

    Every component should be documented through clear naming and clean internal structure.

    Readable code is preferred over clever code.

    ---

    # QS-019

    Placeholder content must be obvious.

    Use meaningful placeholder text.

    Never leave empty visual elements.

    ---

    # QS-020

    Version 1 components should prioritize stability over configurability.

    Do not add configuration options that are not currently needed.

    ---

    # QS-021

    No component may import another component simply to reproduce an existing layout.

    Reuse through composition, not duplication.

    ---

    # QS-022

    Future extensibility should be considered, but speculative features should not be implemented.

    Architecture should be open.

    Implementation should remain minimal.

- unnecessary nesting

# QS-025

Acceptance Criteria

A reusable component is considered production-ready only if:

- It contains no business logic.
- It receives configurable data through props.
